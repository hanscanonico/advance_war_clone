Grid Commander · Balance instrument · Plan for review

Balance Lab — any map, any commanders, any tier, full telemetry

One headless tool that pits two independent AIs against each other on any shipped map, each side carrying any commander at any difficulty tier, over N seeded matches with both seats swapped — and records not just who won, but a turn-by-turn timeline: each side's funds, what it built, what it killed, what it lost, every capture and every power. Win-rate summaries flag what's unbalanced; the timeline shows why. And any matchup the Lab can score it can also show: the same spec and seed boot the real game window with both sides AI-driven, so a suspicious row in the report becomes a battle you watch live.

1 What exists, what's missing

This is not a from-scratch build. tools/run_commander_balance.gd already plays deterministic AI-vs-AI headlessly, side-swaps seats, ranks day-cap games on the Advance Wars timed-match tiebreak, and writes CSV/JSON with win-rate bands — it's the committed instrument behind two standing documents (docs/commander_balance.md, docs/difficulty_check.md). But its two modes each fix a dimension the Balance Lab must free:

What this instrument immediately unlocks

Two standing debts are waiting for exactly this tool: the DF4 difficulty ladder must be re-run after the planner fixes (docs/difficulty_check.md §4 says its numbers describe a superseded planner and a superseded board), and the +14.9 pp first-side bias accepted during the air/naval work belongs to a base-game balance pass that needs per-map bias numbers. 2 Architecture — the sim is already a simulator

The simulation/presentation split does all the heavy lifting: core/ and ai/ are Node-free, GameState owns a seedable RNG, and the AI is driven one command at a time by whoever hosts it. A headless match is ~15 lines, and the per-side-planner shape already exists in the runner's _play_tiers (tools/run_commander_balance.gd:714): 

# the whole match loop — MatchEngine generalizes exactly this
var map := MapData.load_from_file(map_path, terrain_db)   # any maps/*.txt
var state := GameState.create(map, unit_db, chart)
state.rng.seed = seed_val                                  # deterministic
state.set_commander(1, commander_db.by_id(red.commander))
state.set_commander(2, commander_db.by_id(blue.commander))
var planners := {                                          # two independent AIs
	1: AIController.new(unit_db, difficulty_db.by_id(red.tier).profile()),
	2: AIController.new(unit_db, difficulty_db.by_id(blue.tier).profile()),
}
while state.winner == 0 and state.day <= days_cap and commands < COMMAND_CAP:
	var command: Command = planners[state.current_team].plan_next_command(state)
	if command.validate(state) != "":                  # planner/rules disagreement = hard failure
		rejected += 1
		command = EndTurnCommand.new()
	recorder.before_apply(state, command)               # NEW: telemetry hook (harness-side)
	command.apply(state)
	recorder.after_apply(state, command)
	commands += 1

Zero changes to the game rules

All measurement lands under tools/balance/. The recorder observes: it inspects each command before/after apply and reads public GameState fields. No signals added to core/, no new hooks in ai/ — the thing being measured is bit-for-bit the shipped game. The one change outside tools/ is presentation: the battle scene learns to host a planner per team for watch mode (BS3).

Determinism inherited

Same map + seed + side specs ⇒ byte-identical CSV rows and timeline, because the AI is lookahead-free and RNG-free and only CombatResolver draws from the seeded state.rng. A tuning change is attributable to the change, never to noise.

Scoring rules shared, verbatim

Day-cap games rank by the existing tiebreak — properties, then units, then funds — with the same termination labels (rout/hq/day_cap/command_cap) and the same hard invariants (0 rejected commands, 0 cap stalls). Balance Lab numbers stay comparable with both standing docs.

3 The tool — one matchup spec, three sweep axes

A side is <commander>:<tier> — commander id or none, tier easy/normal/hard, defaults none:normal. Every seed is played both seats (the pairing the existing runner already uses), so a first-move advantage cancels out of win rates and is reported separately as bias.

# one matchup, full telemetry — "why does Gideon crush Cass here?"
make balance-sim SIM="--map=ironworks --red=gideon_holt:normal --blue=cass_orlov:normal --seeds=10"

# mixed tiers and doctrines — "is Difficult worth a commander handicap?"
make balance-sim SIM="--map=scrimmage --red=cass_orlov:hard --blue=gideon_holt:easy --seeds=15"

# sweep axis 1 — every commander (vs neutral) at one tier on one board
make balance-sim SIM="--map=the_straits --sweep=commanders --tier=normal --seeds=4"

# sweep axis 2 — map fairness: identical mirror on every shipped board
make balance-sim SIM="--sweep=maps --red=none:normal --blue=none:normal --seeds=6"

# sweep axis 3 — the DF4 ladder, now with doctrines allowed on both sides
make balance-sim SIM="--sweep=tiers --map=scrimmage --commander=alina_ward --seeds=15"

# watch a match from the report live in the game window — same spec + seed = the same battle
make balance-watch SIM="--map=ironworks --red=gideon_holt:normal --blue=cass_orlov:normal --seed=1003"

Remaining flags carry over from the existing runner: --seeds=, --days= (default 20), --out=. Output goes to reports/balance_sim/<run-name>/ (gitignored like all reports): matches.csv — one row per match; timeline.csv — one row per side per day, keyed by match_id; summary.json — aggregates and flags (§5, BS2).

4 The timeline — what gets recorded every turn

One timeline.csv row per played turn (match × day × side). A row covers everything from that side's income tick to its EndTurnCommand, so every event in the game lands in exactly one row:

Column groupColumnsDetail
Key match_id · day · team · commander · tier Joins to matches.csv for map, seed, seats, outcome.
Money funds_start · income · spent · funds_end funds_start is read after the income/upkeep tick of TurnRules.begin_turn; spent is the sum of applied BuildCommand costs, so funds_start − spent = funds_end reconciles (modulo commander economy hooks, which then become visible as the residual).
Production built · built_value Unit ids with counts (tank;infantry×2) and their summed cost — the build order, turn by turn.
Combat killed · lost · killed_value · lost_value Enemy units destroyed this turn / own units destroyed this turn (counter-fire and power effects included), as unit-id lists plus summed cost. Attribution is by command, not blind diffing — §D3.
Board unit_count · army_value · properties · captures End-of-turn strength (army_value = Σ cost × HP fraction — the honest number, since a 2 HP tank isn't a tank), property count, captures completed this turn.
Powers power_charge · power_fired Meter percentage at turn end; whether the Command Power fired this turn.
Cost commands · planning_ms Commands issued and AI planning time — keeps the R3 turn-time watch item measurable for free on every run.

Reading it is deliberately boring: filter one match_id in any spreadsheet and you watch the game — day 3 Blue banks for a bomber, day 5 Red's army value collapses to a power spike, day 8 the property lines cross. That's the "why" behind every win-rate number in summary.json.

BS2 · Sweeps and the imbalance report done

The three --sweep= axes of §3, and a summary.json that judges: side-normalized win rate per swept value with the standing bands (45–55% preferred, 40–60% warning — same thresholds as docs/commander_balance.md), first-side bias per map, termination mix with a low-confidence flag when day-cap tiebreaks dominate (§R2), and per-1000-funds efficiency from the timeline. Plus docs/balance_sim.md — how to run it, how to read it, what the flags mean.

  • Neutral mirror sweep across all 12 maps produces the per-map bias table (the base-game bias debt gets its numbers)
  • Known outliers reproduce: Gideon Holt / Tomas Reed flag high, Cass Orlov / Rhea Sol low — the instrument agrees with the committed record
  • Full 12-map neutral sweep at 6 seeds finishes in single-digit minutes headless

BS3 · Watch any match live done

The same side-spec grammar boots the real battle scene, windowed, both teams AI-driven: make balance-watch. One presentation-layer change makes it possible — scenes/battle/battle.gd + battle_setup.gd learn to host a planner per team (the harness's planners dict, brought into the scene) instead of today's single shared AIController, and a --seed= flag routes to state.rng.seed instead of randomize(). Because the sim and both planners are deterministic, the windowed run replays the harness's command sequence exactly: pick a strange row in matches.csv, feed its spec and seed back, and watch that very game. Launched through the focus-safe wrapper (tools/godot_gui.sh) like every windowed target.

  • Replay fidelity: a watched spec + seed ends with the same winner on the same day as its harness row
  • A mixed matchup (different commander and tier per side) visibly plans differently per side
  • Still zero changes under core/ and ai/ — the scene hosts planners; the sim and the AI are untouched

BS4 · The reading surface done

A self-contained HTML report generated per run from summary.json + timeline.csv: win-rate heat-map for sweeps, per-map bias chart, and per-match funds / army-value curves — the timeline §4 promises, drawn instead of eyeballed in a spreadsheet. Static file, no server, gitignored with the rest of the report.

  • One command produces the report next to the CSVs it renders
  • Every number in it is read from the run's files — the generator computes nothing the CSVs don't already say

6 Decisions

D1 · Extract the engine; the two shipped gates stay byte-stable

The shared loop moves to tools/balance/; run_commander_balance.gd keeps its CLI and both committed gates (commander matrix, DF4 ladder) as thin presets over the engine. The merge bar is a fixed-seed byte-diff of their reports before/after — the instruments behind two standing documents must not drift while gaining a sibling.

D2 · Telemetry observes; the sim is untouched

No signals, hooks or fields are added under core/ or ai/. The recorder reads public state around each command.apply() — which the harness already brokers. What the Lab measures is therefore exactly what ships, and the tool can never bend the game toward measurability.

D3 · Kill/loss attribution is by command, never blind diffing

A unit vanishing from the board is not always a death: JoinCommand merges two units, LoadCommand embarks one into a transport. The recorder classifies removals by the command that caused them — AttackCommand (both directions: target death and counter-fire death), PowerCommand effects, join-merges, embarks. A reconciliation test (BS1) makes miscounting a red build, not a quiet lie in the data.

D4 · One timeline row per played turn, one CSV per run

Per-turn is the requested grain and it keeps a full run in one spreadsheet-friendly file (~70k rows for the biggest batch imaginable — trivial). A per-command event log — every move and shot with coordinates, i.e. a replay — is deliberately out of scope here and parked as Q3.

D5 · Sweep one axis per run

Commanders × tiers × maps × seeds ≈ 12²·3²·12·N — a six-figure matrix nobody reads. A run pins everything except one swept axis, so every batch answers one question and finishes in minutes. Broad sweeps are a sequence of runs, which the deterministic seeds make exactly reproducible and comparable.

D6 · Tiers stay honest, per the difficulty plan's lock

A side's tier is only which AIProfile plans its moves — the Lab adds no handicap, no income tilt, no friendlier dice at any tier, in either direction (difficulty plan D2/D3). Mixing tier and commander per side is new measurement, not new mechanics.

D7 · Watching is the same match, not a re-creation

Watch mode doesn't record or stream anything from the harness — it simply runs the same deterministic pipeline with animation between commands. That works because the AI plans from state alone (lookahead-free, RNG-free) and only CombatResolver draws from the seeded state.rng; BattleAiRunner's pacing delays are pure presentation and touch no sim state. The replay-fidelity check in BS3 pins the equivalence — if the scene ever consumed the RNG or reordered a command, the watched winner would diverge from the CSV row and the check goes red. One alignment detail rides along: the scene's per-turn command cap and the harness's match-level cap are unified in the shared engine so neither can cut a game the other would let run.

7 Risks

R1 · On naval boards the Lab measures the AI, not the map

The naval plan's standing R1 holds: the AI never plans a ferry, so on isthmus and the_straits its results reflect what the AI can express there, not what humans would find. Mitigation: the summary annotates naval-board rows as AI-bounded rather than hiding them. This is a documented reading rule, not a fix — fixing it is AI work owned by the naval plan.

R2 · Day-cap tiebreak noise can blind a board

docs/difficulty_check.md finding (a) is the caution: ironworks scored the known-weaker Easy even with Normal because day-cap tiebreaks turned over on noise. Mitigation: the summary reports the termination mix and marks any swept value low-confidence when decisive endings (rout/hq) are a minority of its games; --days= stays a flag for probing whether a longer cap resolves a board.

R3 · The refactor could silently move the two shipped gates

Extracting the engine touches the code both standing documents rest on. That's why D1's byte-diff is a merge bar, not a nicety — if the fixed-seed reports differ by one byte, the extraction is wrong, whatever the diff says it "improved".

R4 · Mixed-tier matchups conflate map bias with tier strength

Held in check by construction: every seed plays both seats, win rates are side-normalized, and bias is reported as its own number per map — the same discipline the existing runner already applies to commander pairings.

R5 · Telemetry cost

Recording is reads and appends; matches stay tens-to-hundreds of milliseconds headless. Timeline rows ≈ matches × days × 2 — a few MB of CSV for the largest batch, written buffered. planning_ms lands in every row, so if the recorder ever did get expensive it would be visible in its own output.

8 Validation

Correctness gates

  • Determinism: identical spec ⇒ byte-identical matches.csv and timeline.csv
  • Reconciliation: per match, timeline funds arithmetic closes and built − lost = final army, per side
  • Attribution unit tests: kill vs counter-death vs join vs embark vs power-kill each pinned by a GUT test (the recorder is Node-free precisely so GUT can drive it)
  • Hard invariants inherited: 0 rejected commands, 0 command-cap stalls, or the run exits 1

Instrument sanity

  • Control run: neutral Normal mirror ⇒ ~50% side-normalized, as the difficulty doc's control measured
  • Both shipped gates byte-stable across the extraction (D1)
  • Known commander outliers reproduce on the fixtures (BS2)
  • Runtime budget measured and recorded in docs/balance_sim.md

Like its siblings, the Lab stays out of make verify/make test — it's an opt-in instrument, not a merge gate. Only its own unit tests (recorder attribution, engine determinism) join the suite.

9 Open questions — answered at review

Q1 · Is the HTML report in scope? yes

CSVs + summary flags answer "what's unbalanced" on their own; the report makes the timeline pleasant to read. Ship BS3 with this work, or land BS1–BS2 and decide once you've read a few runs raw?

Q2 · Fold the old CLIs in, eventually? keep them

D1 keeps make commander-balance / make difficulty-check as byte-stable presets. Should they stay forever, or be re-expressed as documented Balance Lab invocations once it has earned trust? (Recommendation: keep the presets — two committed docs cite their exact flags.)

Q3 · Per-command replay log? yes

A finer JSONL — every command with coordinates, effectively a replay file — would let a suspicious match be stepped through move by move. Real value, real scope. Not in this plan unless you want it; the per-turn grain is the committed deliverable.
