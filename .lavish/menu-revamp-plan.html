Menu Revamp — the design system comes home

The main menu is the only screen in the game with no art direction: stock Godot greys on a flat olive rectangle, while the board, the HUD and both cut-ins all speak with one voice. The Grid Commander Design System handoff (handoff/main-menu/ in the Downloads zip) writes that voice down — cream panels, ink borders, hard offset shadows, pixel type, a live terrain field breathing behind everything — and its palette was lifted from this game's own atlases, so adopting it is alignment, not invention. This plan adapts the mockup to the game as it actually ships: 12 maps, not 2; four speed tiers, not three; commanders, factions, and a rule that "Red"/"Blue" never reaches a player's screen.

What the handoff hands us, and what stands today

The handoff inventory

SPEC.md (the screen, element by element), a working React mockup (Main Menu.html), five token sheets (colors / effects / fonts / spacing / typography), three reference components (Button / Panel / FactionTag — source to read, not code to port), and a folder of 32px terrain sprites the mockup previews with. Source: ~/Downloads/Grid Commander Design System (5).zip — every number this plan needs is quoted here, so the repo never depends on the zip.

The menu today

scenes/menu/main_menu.tscn is a centered VBoxContainer of stock controls — an OptionButton each for map, difficulty and speed, two CheckButtons, four buttons — on a flat Color(0.10, 0.14, 0.10) backdrop. No Theme resource, no font file, no StyleBox exists anywhere in the project; the entire game runs on Godot's default font. The commander select panel (commander_select_panel.gd, built in code, opened over the menu) and the battle HUD widgets each carry their own inline styleboxes around the shared CommanderVisuals constants.

Layout sketch in the handoff's own tokens and web fonts (illustrative — the game draws this with vendored fonts, real tile-grid thumbnails, and the slow-panning terrain field behind it at 14% opacity).

The mockup's bones survive whole: header with icon and wordmark, a Match Setup panel beside an action stack, the blinking PRESS START. Five places the mockup and the shipped game disagree, and in every one the game wins:

12 maps, not 2 → a scrollable two-up grid

The mockup shows two thumbnails side by side; the roster (core/map_catalog.gd, smallest board first) ships twelve. The picker becomes a ScrollContainer holding the mockup's two-up thumbnail cells — selected map gets the raised surface, 3px red border and "✓" exactly per spec, and the panel header-right shows name · size. Each cell keeps the current dropdown's tooltip (size, property count, description) so no information is lost in the trade.

Four speed tiers, not three — and it stays a device preference

The segmented control reads GameSpeed.ordered() — Slow · Normal · Quick · Instant — and difficulty reads DifficultyDB's three tiers. Segment labels are populated from those authorities, never typed in. Speed keeps the game-speed plan's D1 semantics untouched: tapping a segment writes Settings.set_speed immediately and never rides MatchConfig. Battle animations likewise stays the standing Settings preference it already is.

No "Red vs Blue" — the menu speaks faction

The mockup's chips say "Red · P1 / Blue · AI" and its tagline says "RED VS BLUE" — both illegal under faction-identity D5, which this repo already enforces everywhere else. Chips render the default side identities (Meridian · P1, Aurora · AI/P2) through CommanderVisuals, and the tagline drops the faction words. The 1P/2P button hues stay — colors are not words — but they are read from the faction theme table, not re-declared.

Continue: disabled, not hidden — and the flows do not move

Today ContinueButton vanishes without a save; the spec shows it present-but-disabled (45% opacity, desaturated), which teaches that continuing is a thing this menu does. That is the only behavior change in the whole plan. Everything else rewires nothing: 1P/2P still open the commander select, confirm still walks _start → MatchConfig → battle.tscn, and the --co-select capture hooks keep working.

Starting numbers — the ÷2 rule

The mockup is authored for a ~1280px viewport; the game canvas is 640×360 stretched 2× into the default 1280×720 window (project.godot, untouched). So every handoff pixel maps to canvas at ÷2 and renders at mockup size in the default window. Rounded to whole canvas pixels, favouring the chunkier reading:

3

Decisions

D1 · Where do the tokens live? proposed

In one new presentation authority, scenes/common/ui_theme.gd (class_name UiTheme): the shell palette (slates, neutrals, capture green), the two vendored fonts, and stylebox factories for the recipes — cream panel, dark panel, hard shadow, faction button, segment, checkbox, focus ring. Where a color already has an authority, UiTheme re-exports it rather than re-declaring it: faction colors stay CommanderVisuals.FactionTheme's, cream and ink stay CommanderVisuals.PAPER / PAPER_INK / HARD_BORDER. Not a .tres Theme — this repo's whole UI is code-built styleboxes (CommanderSelectPanel._flat/_hard, commander_card._flat_box…) and a hand-edited Theme resource is the one format nobody can review in a diff. Not data/ — data/ is for numbers the sim reads (the game-speed plan's D2 precedent), and the sim must never read a menu color.

D2 · Typography: vendor the two faces, for real proposed

Pixelify Sans (display & UI chrome) and Silkscreen (micro-labels, numerals) land as TTFs under assets/fonts/, recorded in assets/LICENSES.md — both SIL OFL, exactly the diligence the project's legal note demands. The handoff calls them "chosen substitutes" because the repo had no font; the substitution ends here — they become the game's faces. Antialiasing off, sizes on the ÷2 grid, so they rasterize on the pixel grid the rest of the art lives on. The handoff's Google-Fonts @import cannot ship in an engine build; vendoring is the only option that works offline and in export.

D3 · The resolution does not move; the shadow is a StyleBox trick proposed

The 640×360 canvas and canvas_items stretch are the board's contract — the battle scene is composed against them — so the menu adapts to the canvas, never the reverse (§2's ÷2 rule). The signature hard shadow needs no custom drawing: StyleBoxFlat with shadow_size = n, shadow_offset = (n, n) and antialiasing off grows the shadow rect by n then shifts it — flush under the top-left edges, exactly 2n of hard ink past the bottom-right — reproducing CSS 2n 2n 0 with zero blur. Verified visually in MN1 before anything builds on it. The press-down interaction (button translates 2px, shadow collapses) is the pressed stylebox plus a content-margin shift — state swap, no tween.

D4 · Identity on the menu speaks faction proposed

Faction-identity D5 is already the law: "Red"/"Blue" survive as developer slot vocabulary only. The mockup predates that rule, so its literals are corrected at the door: chips say Meridian · P1 / Aurora · AI (or P2) in the classic default hues, the tagline becomes faction-silent ("TURN-BASED TACTICS · PICK YOUR GROUND" as the starting text), and every faction hue on the screen — button fills, chip dots, selected map border — resolves through CommanderVisuals so a future palette change lands here for free. The menu shows the default identities deliberately: commanders are picked after 1P/2P, and the defaults are exactly what a commander-less match plays as.

D5 · One renderer draws every board pixel on the menu proposed

A new scenes/menu/map_thumbnail.gd — one Control, one canvas item — draws a MapData by blitting assets/tiles/terrain_atlas.png regions per cell: column from TerrainType.atlas_col, row from SideIdentity.atlas_row — the exact authorities battle_view.gd paints the board with. No symbol table, no per-terrain color list, no second opinion (the movement-resolver lesson, again). The panning backdrop is the same renderer's output baked once to an ImageTexture, tiled and scrolled at 14% opacity behind the whole screen. Owned properties draw in the default meridian/aurora rows, so a thumbnail is a truthful miniature of day 1.

D6 · Scope: the menu flow, whole — the battle HUD, not at all proposed

The revamp covers everything a player sees between boot and the first turn: the menu screen and the commander select panel that opens over it (an alignment pass — its dark palette already sits a shade from the slate tokens; it gains the fonts, the hard shadows and the shared constants, and keeps its layout, its gold selection highlight and its begin/confirmed/cancelled API untouched). The battle HUD keeps its current dress: it already shares CommanderVisuals, nothing forks, and its alignment to UiTheme is a named follow-up rather than silent drift. Restyling half the game's HUD as a rider on a menu plan is how scope creep — the named top risk of the base plan — happens.

Milestones — each one shippable

MN1 · The system lands: tokens, fonts, and the dressed screen size M

After MN1 the menu is the mockup's layout in the mockup's dress — with the map picker still the styled dropdown for one more milestone, so the scene rebuild and the thumbnail renderer never land in the same diff.

  • scenes/common/ui_theme.gd — shell palette (new tokens only; re-exports for everything that has an authority), stylebox factories (panel, button×4 variants×states, segment, checkbox, focus ring), font loaders. The D3 shadow trick proven here first.
  • assets/fonts/ — Pixelify Sans + Silkscreen TTFs, antialiasing off; assets/LICENSES.md gains both OFL entries.
  • main_menu.tscn rebuilt to the two-column layout: header (icon, wordmark with ink offset shadow, faction-silent tagline), Match Setup panel (dropdown map row for now, segmented Difficulty/Speed from their DBs, Fog/Animations checkboxes), action stack (1P meridian / 2P aurora with Silkscreen suffixes, Continue disabled-until-save, ghost Quit, identity chips, blinking PRESS START).
  • main_menu.gd keeps every flow and hook: Settings speed/animations writes, _start → MatchConfig, --co-select capture, focus opens on 1 Player; segmented controls and checkboxes are keyboard/gamepad navigable with the chunky focus ring.
  • Proof: make run at 2× and at a 640×360 window — every label legible, every control reachable without a mouse; make verify green (nothing under core/ or ai/ changes in this plan, ever).

MN2 · The board comes to the menu size M

The dropdown dies; the roster becomes visible terrain.

  • scenes/menu/map_thumbnail.gd — D5's renderer: one canvas item per map, atlas columns × identity rows, sized to its cell (4–5px canvas tiles for a 20×15 board).
  • The scrollable two-up grid replaces MapOption: selected cell gets raised surface + meridian border + ✓, panel header-right reads name · size, every cell carries the old tooltip's facts. Scroll position follows keyboard focus; twelve maps, smallest first, exactly MapCatalog.ordered().
  • The panning backdrop: one shipped board baked to an ImageTexture, tiled 3×3, drifting −64px canvas diagonally over 40s at 14% opacity behind a radial slate highlight. Pinned still whenever ScreenshotUtil.requested() — the capture must not depend on when it was taken (the animator's capturing precedent), and the PRESS START blink pins with it.
  • Proof: make run — flip through all twelve thumbnails against the boards they launch; a --co-select capture is byte-stable across two runs.

MN3 · The select panel joins the system size S

The alignment pass of D6, plus the polish debt of MN1–MN2.

  • commander_select_panel.gd swaps its private palette for UiTheme tokens (its slate was already a shade away), gains the two fonts and the hard shadows; layout, gold selection highlight and public API untouched.
  • Press-down (2px translate, shadow collapse), hover brightness, disabled desaturation land on every button through the shared factories — one implementation, both screens.
  • The --co-select gallery captures re-baselined once, deliberately, in their own commit — the diff is the restyle, reviewed as such.
  • README gains the design-system note: where the tokens live, where the fonts came from, what the handoff zip was.
  • Proof: make run through menu → select → battle and back; the seam between the two screens reads as one game. make verify green, still zero core//ai/ lines touched.
5

Risks

R1 · Pixel type at 640×360 stops being readable

The named top risk. Silkscreen at 6px canvas is 12 physical pixels on the default window and 6 in a 1× window; a fractional OS scale could smear either face into mud. Mitigation is mechanical: antialiasing off, sizes only on the ÷2 grid, and MN1's gate explicitly plays the menu at 1× and 2× before building further. If a label fails at 1×, the size table moves up a step — the mockup is a reference, not a contract, and legibility outranks fidelity.

R2 · The thumbnail becomes a second opinion on terrain

A renderer with its own symbol map or color table would drift from the board the moment a terrain is added — the range-overlay-vs-command bug, relocated. D5 closes it by construction: the thumbnail reads TerrainType.atlas_col and SideIdentity.atlas_row — the board's own authorities — and the backdrop reuses the thumbnail's output. There is nothing to keep in sync because there is only one of it.

R3 · The menu is a capture surface, and now it moves

The --co-select captures shoot the select panel over the live menu; a drifting backdrop or mid-blink PRESS START makes those shots machine-dependent. MN2 pins both under ScreenshotUtil.requested() — the same gate the animator's capturing flag set the precedent for — and the MN3 re-baseline is one deliberate commit, not a flaky diff that gets waved through.

R4 · The third palette

Menu on UiTheme, select panel half-migrated, HUD on raw CommanderVisuals — three dialects where there were two. Contained by D1's re-export rule (UiTheme aliases the shared constants, so there is still exactly one value per color) and by D6 naming the HUD follow-up out loud. The review rule after MN1: no new inline hex/Color literal in scenes/menu/ — every color is a UiTheme or CommanderVisuals name.

R5 · Determinism, saves, sim safety

Structurally safe rather than carefully safe, again: this plan touches scenes/ and assets/ only. MatchConfig carries the same fields, Settings the same keys, the save format is untouched, and the single behavior change (Continue disabled vs hidden) cannot reach a sim. The GUT suite neither gains nor loses a test — per the testing doctrine, presentation is verified by playing it.

6

Explicitly out of scope

The battle HUD & in-battle menus

D6's line. The terrain panel, HUD chips, banners and battle menus keep their dress; aligning them to UiTheme is its own future plan with its own capture re-baselines.

Porting the React components

Button/Panel/FactionTag are read for their numbers and states, then closed. The Godot implementation is styleboxes and containers; no JSX concept survives the crossing.

Menu audio, music, and transitions

No boot jingle, no hover blips, no screen-to-screen wipes beyond what exists. The spec's motion budget (90–240ms, mechanical) is honored inside controls only.

Resolution, stretch, or a settings screen

640×360 canvas_items/keep is the board's contract and stays. No options screen rides along — the menu's five controls are the options, exactly as today.
